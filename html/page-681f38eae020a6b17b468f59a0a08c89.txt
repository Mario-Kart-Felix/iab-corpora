<h2>
<a name="intro">Introduction</a>
</h2> 
 
  <p>This document provides basic guidelines to isolate and resolve many
	 common Ethernet auto-negotiation issues. This document provides a general
	 description of auto-negotiation, and explains the procedure to configure and
	 verify auto-negotiation on Catalyst switches that run the Catalyst OS (CatOS)
	 system software. </p> 
  <p>This document also shows an example of why the most common
	 duplex-mismatch error occurs, and describes how to configure and verify
	 auto-negotiation on Catalyst switches that run Cisco
	 IOS<sup>Â®</sup> System Software.</p> 
   
	 <p>
<b>Note:&nbsp;</b>The new Catalyst switches/modules, such as the Catalyst 6500/6000,
		4500/4000, 3550, and 2950, support 10/100/1000 Mbps negotiated Ethernet
		interfaces or ports. These ports work on 10 Mbps, 100 Mbps, or 1000 Mbps speed
		based on their connection to the other end. These 10/100/1000 Mbps ports can be
		configured for speed and duplex negotiation similar to 10/100 Mbps ports on
		CatOS or Cisco IOS Software-based switches. Therefore, the configurations
		described in this document for 10/100 Mbps port negotiation apply to
		10/100/1000 Mbps ports as well.</p>

<h2>
<a name="when">When to Use Ethernet 10/100 Mb Auto-Negotiation</a>
</h2> 
 
  <p>Auto-negotiation is an optional function of the IEEE 802.3u Fast
	 Ethernet standard that enables devices to automatically exchange information
	 over a link about speed and duplex abilities. </p> 
  <p>Auto-negotiation is targeted at ports. These ports are allocated to
	 areas where transient users or devices connect to a network. For example, many
	 companies provide shared offices or cubes for Account Managers and System
	 Engineers to use when they are in the office. Each office or cube has an
	 Ethernet port permanently connected to the office network. Because it might not
	 be possible to ensure that every user has either a 10 Mb, a 100 Mb Ethernet, or
	 a 10/100 Mb card in their laptop, the switch ports that handle these
	 connections must be able to negotiate their speed and duplex mode. The
	 alternative is to provide both a 10 Mb and a 100 Mb port in each office, or
	 cube and label them accordingly. </p> 
  <p>One of the most common causes of performance issues on 10/100 Mb
	 Ethernet links occurs when one port on the link operates at half-duplex while
	 the other port operates at full-duplex. This occurs when one or both ports on a
	 link are reset and the auto-negotiation process does not result in both link
	 partners having the same configuration. It also can occur when users
	 reconfigure one side of a link and forget to reconfigure the other side. Both
	 sides of a link should have auto-negotiation on, or both sides should have it
	 off. Cisco recommends to leave auto-negotiation on for those devices compliant
	 with 802.3u. </p> 
  <p>Many performance-related support calls are avoided if you correctly
	 configure auto-negotiation. Many Catalyst Ethernet switching modules support
	 10/100 Mb and half-duplex or full-duplex. Exceptions include the Ethernet Group
	 switch modules. The <b>show port capabilities
	 {<span style="font-weight:normal; font-style:italic">mod_num</span>} |
	 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}</b> command shows if the
	 module you are working on supports 10/100 Mb and half-duplex or full-duplex.
	 This document uses two WS-X5530 Supervisor Engine IIIs, each with two optional
	 uplink 10/100 BaseTX Ethernet ports installed. </p> 
   
	 <p>
<b>Note:&nbsp;</b>When the WS-6748-GE-TX module is connected to a network tap device,
		automatic negotiation does not work. In order to resolve this issue, you must
		configure auto-negotiation manually. Go to the interface mode and execute this
		command:</p> 
	 <blockquote>
<pre>Cat6K-IOS(config-if)#<b>speed auto</b>
</pre>
</blockquote> 
  
  
<h2>
<a name="gb">When to Use Ethernet 1000 Mb Auto-Negotiation</a>
</h2>
 
  <p>Basically auto-negotiation in GigabitEthernet covers these
	 items:</p> <ul> 
  <li> 
	 <p> Duplex settings&#8212;While Cisco devices only support full-duplex, the
		IEEE 802.3z standard does have support for half-duplex GigabitEthernet. Because
		of this, duplex is negotiated between GigabitEthernet
		devices.</p>
</li> 
  <li> 
	 <p>Flow Control&#8212;Becuase of the amount of traffic that can be generated
		by GigabitEthernet, there is a PAUSE functionality built into GigabitEthernet.
		The PAUSE frame is a packet that tells the far-end device to stop the
		transmition of packets until the sender is able to handle all the traffic and
		clear its buffers. The PAUSE frame has a timer included, which tells the
		far-end device when to start to send packets again. If that timer expires
		without getting another PAUSE frame, the far-end device can then send packets
		again. Flow-Control is an optional item and must be negotiated. Devices can be
		capable of sending or responding to a PAUSE frame, and they possibly do not
		agree to the flow-control request of the far-end neighbor.</p>
</li> 
  <li> 
	 <p> Negotiation&#8212;Usually built-in Gigabit Ethernet ports are capable of
		negotiation, but in cases like modular SFP or GBIC types, they do not
		negotiate. Line protocol will be down for a Gigabit Ethernet port when
		connected to a Fast Ethernet port. This can be verified via the
		<b>show inter gig4/3 capabilities</b> command: </p> 
	 <blockquote>
<pre>Switch-A#<b>show interface Gig4/3 capabilities</b>
GigabitEthernet4/3
Model                    WS-X4516-10GE-Gbic
Type                     1000BaseT
<b>Speed                    1000
Duplex                   full
</b>
</pre>
</blockquote>
</li>
</ul> 
  <p>Assume that there are two devices, A and B. Assume that each device can
	 have Autonegotiation enabled, or disabled. The correct behaviour of link status
	 with autonegotiation in accordance to the IEEE Std 802.3z-1998 should be like
	 this:</p>
<ul> 
  <li> 
	 <p>If A is enabled and B is enabled, then link status should be reported
		on both devices as link up.</p>
</li> 
  <li> 
	 <p>If A is disabled and B is enabled, then A should report link up and B
		should report link down.</p>
</li> 
  <li> 
	 <p>If A is enabled and B is disabled, then A should report link down and
		B should report link up.</p>
</li>
</ul> 
  <p>By default, all devices are supposed to perform autonegotiation. 802.3z
	 does not specifically define a way to turn Autonegotiation off, for both
	 1GigabitEthernet and 10GigabitEthernet.</p>

<h2>
<a name="cfg_ts">Configure and Troubleshoot Ethernet 10/100Mb Auto-Negotiation on CatOS Switches</a>
</h2> 
 
  <p>This section of the document explains how to examine the behavior of an
	 10/100 Mb Ethernet port that supports auto-negotiation. This procedure shows
	 how to make changes to its default behavior and how to restore it to the
	 default behavior. Complete these steps:</p>
<ol type="1"> 
  <li> 
		<p>Connect the two switches together. </p> 
	  
		<p>Ethernet port 1/1 on switch A is connected to Ethernet port 1/1 on
		  switch B using an Ethernet crossover cable. See <a href="#appb">Appendix
		  B</a> for more information on crossover cables.</p> 
		 
		  <p>
<b>Note:&nbsp;</b>Your actual <i>mod_num/port_num</i> numbers might
			 be different.</p> 
		 
	  
  </li> 
  <li> 
		<p>Examine the capabilities of the ports. </p> 
	  
		<p>The <b>show port capabilities 1/1</b> command
		  displays the capabilities of an Ethernet 10/100 BaseTX 1/1 port on switch A.
		  </p> 
		<p>Issue this command for both of the ports you are troubleshooting.
		  Both ports must support the speed and duplex capabilities if they are supposed
		  to use auto-negotiation. </p> 
		<p>The bold text in this output shows where the information on the
		  speed and duplex mode capabilities are found. </p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>show port capabilities 1/1</b>
Model                    WS-X5530
Port                     1/1
Type                     10/100BaseTX
<b>Speed                    auto,10,100
Duplex                   half,full
</b>
</pre>
</blockquote> 
	  
  </li> 
  <li> 
		<p>Configure auto-negotiation for port 1/1 on both switches.
		  </p> 
	  
		<p>Issue the <b>set port speed 1/1 auto</b>
		  command to configure auto-negotiation for both speed and duplex mode on port
		  1/1 of both switches. Auto is the default for ports that support
		  auto-negotiation. </p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>set port speed 1/1 auto</b>
Port(s) 1/1 speed set to auto detect.
Switch-A (enable)</pre>
</blockquote> 
		 
		  <p>
<b>Note:&nbsp;</b>The <b>set port speed
			 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} auto</b> command also sets
			 the duplex mode to auto. There is no <b>set port duplex
			 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} auto</b> command. </p> 
		 
		<p>Also, this command is redundant because the configurations of the
		  switches had been cleared to their defaults before starting this procedure. The
		  default behavior for Ethernet 10/100 BaseTX ports is auto-negotiation. </p> 
	  
  </li> 
  <li> 
		<p>Determine if the speed and duplex mode are set to auto-negotiate.
		  </p> 
	  
		<p>Issue the <b>show port 1/1</b> command to
		  display the status of ports 1/1 on switches A and B. </p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>show port 1/1</b>
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal  <b>a</b>-full <b>a</b>-100 10/100BaseTX


Switch-B&gt; (enable) <b>show port 1/1</b>  
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- -----------
 1/1                     connected  1          normal <b>a</b>-full  <b>a</b>-100 10/100BaseTX</pre>
</blockquote> 
		<p>The bold text in the preceding output shows where the information
		  on the current status of a port can be found. Most of the normal output from
		  the <b>show port
		  {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}</b> command is omitted. See
		  <a href="#appc">Appendix C</a> for further explanation of the fields
		  in the output of this command. </p> 
		<p>The <b>a</b> prefixes on the full and 100 indicate that
		  this port is not hard coded (configured) for a specific duplex mode or speed.
		  Therefore, it auto-negotiates the duplex mode and speed if the device it is
		  connected to also auto-negotiates duplex mode and speed. </p> 
		<p>The status is connected on both ports, which means that a link
		  pulse is detected from the other port. The status can be connected even if
		  duplex is incorrectly negotiated or incorrectly configured. </p> 
	  
  </li> 
  <li> 
		<p>Change the speed on port 1/1 in switch A to 10 Mb.
		  </p> 
	  
		<p>In order to demonstrate what occurs when one link partner
		  auto-negotiates and the other link partner does not, the speed on port 1/1 in
		  switch A must be set to 10 Mb. Issue the <b>set port speed 1/1
		  10</b> command to set this speed.</p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>set port speed 1/1 10</b>
Port(s) 1/1 speed set to 10Mbps.
Switch-A&gt; (enable)</pre>
</blockquote> 
		 
		  <p>
<b>Note:&nbsp;</b>Hard coding the speed on a port disables all auto-negotiation
			 functionality on the port for speed and duplex. </p> 
		 
		<p>When a port is configured for a speed, the duplex mode is
		  automatically configured for the mode it previously negotiated. In this case,
		  the mode is full-duplex. Therefore, the <b>set port speed 1/1
		  10</b> command causes the duplex mode on port 1/1 to be configured as
		  if the command <b>set port duplex 1/1 full</b> is also
		  issued. This is explained in step 6. </p> 
	  
  </li> 
  <li> 
		<p>Understand the meaning of the <b>a</b> prefix on the
		  duplex and speed status fields. </p> 
	  
		<p>The absence of the <b>a</b> prefix in the status fields
		  of the output from the <b>show port 1/1</b> command on
		  switch A shows that the duplex mode is configured for full and the speed is
		  configured for 10. </p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>show port 1/1</b>
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal  full  10    10/100BaseTX</pre>
</blockquote> 
	  
  </li> 
  <li> 
		<p>View the duplex status of port 1/1 on switch B.
		  </p> 
	  
		<p>The <b>show port 1/1 </b> command on switch B
		  indicates that the port operates at half-duplex and 10 Mb. </p> 
		<blockquote>
<pre>Switch-B&gt; (enable) <b>show port 1/1</b>
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal a-half a-10  10/100BaseTX</pre>
</blockquote> 
		<p>This step shows that it is possible for a link partner to detect
		  the speed at which the other link partner operates, even though the other link
		  partner is not configured for auto-negotiation. In order to detect the speed,
		  the link partner senses the type of electrical signal that arrives and sees if
		  it is 10 Mb or 100 Mb. This is how switch B determines that port 1/1 operates
		  at 10 Mb. </p> 
		<p>It is not possible to detect the correct duplex mode in the same
		  method that the correct speed can be detected. In this case, where the 1/1 port
		  of switch B is configured for auto-negotiation and the 1/1 port of switch A is
		  not, the 1/1 port of switch B is forced to select the default duplex mode. On
		  Catalyst Ethernet ports, the default mode is auto-negotiate. If
		  auto-negotiation fails, the default mode is half-duplex. </p> 
		<p>This example also shows that a link can be successfully connected
		  when there is a mismatch in the duplex modes. Port 1/1 on switch A is
		  configured for full-duplex while port 1/1 on switch B is defaulted to
		  half-duplex. Configure both link partners to avoid this. </p> 
		<p>The <b>a</b> prefix on the Duplex and Speed status fields
		  does not always mean that the current behavior is negotiated. Sometimes it can
		  mean that the port is not configured for a speed or duplex mode. </p> 
		<p>The previous output from switch B shows duplex as a-half and speed
		  as a-10. This indicates that the port operates at 10 Mb in half-duplex mode. In
		  this example, however, the link partner on this port (port 1/1 on switch A) is
		  configured for full and 10 Mb. Therefore, it is not possible for port 1/1 on
		  switch B to auto-negotiate current behavior. This proves that the
		  <b>a</b> prefix only indicates a willingness to perform
		  auto-negotiation, and not that auto-negotiation actually took place. </p> 
	  
  </li> 
  <li> 
		<p>Understand the duplex mismatch error. </p> 
	  
		<p>This message about a duplex mode mismatch is displayed on switch A
		  after the speed on port 1/1 is changed to 10 Mb. This mismatch occurs because
		  the 1/1 port of switch B defaults to half-duplex when it senses its link
		  partner no longer performs auto-negotiation. </p> 
		<blockquote>
<pre>%CDP-4-DUPLEXMISMATCH:Full/half-duplex mismatch detected o1</pre>
</blockquote> 
		<p>It is important to note that this message is created by the Cisco
		  Discovery Protocol (CDP), not the 802.3 auto-negotiation protocol. CDP can
		  report problems it discovers, but it typically does not automatically fix them.
		  </p> 
		<p>A duplex mismatch might or might not result in an error message.
		  Another indication of a duplex mismatch is the rapid increase of FCS and
		  alignment errors on the half-duplex side, and runts on the full-duplex port.
		  This can be seen in a <b>show port
		  {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}</b> command. </p> 
	  
  </li> 
  <li> 
		<p>Understand the spanning tree error messages. </p> 
	  
		<p>In addition to the duplex mismatch error message in step 8, you
		  might also see these spanning tree messages when you change the speed on a
		  link. </p> 
		<blockquote>
<pre>%PAGP-5-PORTFROMSTP:Port 1/1 left bridge port 1/1
%PAGP-5-PORTTOSTP:Port 1/1 joined bridge port 1/1</pre>
</blockquote> 
		 
		  <p>
<b>Note:&nbsp;</b>Refer to
			 <a href="http://www.cisco.com/en/US/tech/tk389/tk621/technologies_configuration_example09186a008009467c.shtml">Understanding
			 and Configuring Spanning Tree Protocol (STP) on Catalyst Switches</a> for
			 more information on spanning tree.</p> 
		 
	  
  </li> 
  <li> 
		<p>Change the duplex mode to half on port 1/1 on switch A.
		  </p> 
	  
		<p> Issue the <b>set port duplex 1/1 half</b>
		  command to set the mode on port 1/1 in switch A to half. This demonstrates what
		  occurs when the duplex mode is configured.</p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>set port duplex 1/1 half</b>
Port(s) 1/1 set to half-duplex.
Switch-A&gt; (enable)</pre>
</blockquote> 
		<p>The <b>show port 1/1</b> command displays the
		  change in the duplex mode on this port. </p> 
		<blockquote>
<pre>Switch-A&gt; (enable) <b>show port 1/1</b>
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal   half    10 10/100BaseTX</pre>
</blockquote> 
		<p>At this point, ports 1/1 on both switches operate at half-duplex.
		  Port 1/1 on switch B, however, is still configured to auto-negotiate, as
		  displayed in this output of the <b>show port 1/1</b>
		  command. </p> 
		<blockquote>
<pre>Switch-B&gt; (enable) <b>show port 1/1</b>
Port  Name               Status     Vlan       Level  Duplex Speed Type
----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal a-half a-10  10/100BaseTX</pre>
</blockquote> 
		<p>Step 11 shows how to configure the duplex mode on port 1/1 in
		  switch B to half. This is the recommended policy of always configuring both
		  link partners in the same way. </p> 
	  
  </li> 
  <li> 
		<p>Set the duplex mode and speed of port 1/1 on switch B.
		  </p> 
	  
		<p>This step sets the duplex mode to half and speed to 10 on port 1/1
		  in switch. This implements the policy of always configuring both link partners
		  for the same behavior.</p> 
		<p>This is the output when you issue the <b>set port duplex
		  1/1 half</b> command on switch B. </p> 
		<blockquote>
<pre>Switch-B&gt; (enable) <b>set port duplex 1/1 half</b>
Port 1/1 is in auto-sensing mode.
Switch-B&gt; (enable)</pre>
</blockquote> 
		<p>The <b>set port duplex 1/1 half</b> command
		  fails because this command does not work if auto-negotiation is enabled. This
		  also means that this command does not disable auto-negotiation.
		  Auto-negotiation can only be disabled when you issue the <b>set port
		  speed {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span> {10 | 100}}</b>
		  command. </p> 
		<p>This is the output when you issue the<b> set port speed
		  1/1 10</b> command on switch B. </p> 
		<blockquote>
<pre>Switch-B&gt; (enable) <b>set port speed 1/1 10</b>
Port(s) 1/1 speed set to 10Mbps.
Switch-B&gt; (enable)</pre>
</blockquote>

----- ------------------ ---------- ---------- ------ ------ ----- ------------
 1/1                     connected  1          normal   half    10 10/100BaseTX</pre>
</blockquote> 
		 
		  <p>
<b>Note:&nbsp;</b>The <b>set port duplex
			 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span> {half | full }}</b> command
			 is dependent on the <b>set port speed
			 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span> {10 | 100 }}</b> command. In
			 other words, you must set the speed before you can set the duplex mode.</p> 
		 
	  
  </li> 
  <li> 
		<p>Restore the default duplex mode and speed to <b>ports
		  1/1</b> on both switches. </p>

<b>Note:&nbsp;</b>Once the duplex mode of a port is configured to something other
			 than <b>auto</b>, the only method to configure the port to
			 auto-sense the duplex mode is to issue the <b>set port speed
			 {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} auto</b> command. There is
			 no <b>set port duplex {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}
			 auto</b> command. In other words, issuing the<b> set port
			 speed {<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} auto</b> command has
			 the effect of resetting both port speed sensing and duplex mode sensing to
			 <b>auto</b>.</p> 
		 
	  
  </li>

<h2>
<a name="appb">Appendix B: Ethernet Crossover Cables</a>
</h2> 
 
  <p>Ethernet ports on Catalyst switches have built-in (on-board) Ethernet
	 transceivers. Devices that connect to Ethernet ports might have on-board
	 Ethernet transceivers or use external transceivers. </p> 
  <p>Use a straight-through patch cable, such as a CAT5 10/100BaseT
	 unshielded twisted pair (UTP) patch cable, when you connect a PC, server,
	 printer, or other end-user devices (such as a router) to a switch.
	 Straight-through means that pin 1 on one end of the cable is connected to pin 1
	 on the other end, pin 2 on one end of the cable is connected to pin 2 on the
	 other end, and so forth. </p> 
  <p>Use a crossover cable, such as a CAT5 10/100BaseT UTP crossover patch
	 cable,when you connect another switch port, or other Layer 2 port to an
	 Ethernet port on a switch. In this case, the pins are connected (see
	 <a href="#f1">Figure 1</a> and <a href="#f2">Figure 2</a>).
	 </p> 
  <p>A convenient rule of thumb is to use a crossover cable when the two
	 ports that are connected are in the same layer of the OSI model. If you cross
	 OSI layers, use a straight-through cable. Treat PCs as Layer 3 ports, hubs and
	 most Layer 3 switches as Layer 2 ports. Some devices, especially common on
	 hubs, have a button that can toggle between accepting a straight-through or
	 crossover cable. Therefore, this rule of thumb does not always apply. </p> 
   
	 <p>
<b>Note:&nbsp;</b>Use a crossover cable when you connect two ports in the same layer of
		the OSI model, such as router to router (Layer 3) or switch to switch (Layer
		2). Use a straight-through cable if the two ports are in different layers, such
		as router to switch (Layer 3 to 2) or PC to switch (Layer 3 to 2). For this
		rule, treat a PC as a Layer 3 device. </p>

<b>Fiber Cable Connection Guidelines</b>
</p> 
  <p>If you use an Ethernet port on the switch with a fiber interface to
	 connect to another switch port, a router port, or other Layer 2 device, you
	 need to reverse the connection on one of the devices. Rotate the connector one
	 half turn or cross over the individual fiber connectors to reverse the
	 connection. Think about each fiber as either fiber A or fiber B. If a
	 straight-through connection is A-to-A and B-to-B, a crossover connection is
	 A-to-B and B-to-A. </p>

<h2>
<a name="appe">Appendix E: Frequently Asked Questions</a>
</h2> 
<ol type="1"> 
  <li> 
	 <p>When should you use auto-negotiation? </p> 
	 <p>Cisco recommends that auto-negotiation be used when the devices
		involved are compliant with the 802.3u standard. Refer to
		<a href="http://www.cisco.com/en/US/products/hw/switches/ps708/products_tech_note09186a00800a7af0.shtml">Troubleshooting
		Cisco Catalyst Switches to NIC Compatibility Issues</a> for more information
		on specific products. Auto-negotiation is very useful for ports where devices
		with different capabilities are connected and disconnected on a regular basis.
		A good example is offices that are used by visiting employees who bring their
		own laptops. </p>
</li> 
  <li> 
	 <p>How can you configure a port for auto-negotiation? </p> 
	 <p>Issue the <b>set port speed
		{<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} auto</b> command. This
		resets both the speed and duplex mode to auto-negotiate. On a Catalyst Cisco
		IOS Software platform, issue the interface command <b>speed
		auto</b>. </p>
</li> 
  <li> 
	 <p>How can you tell how your port is configured? </p> 
	 <p>Issue the <b>show port
		{<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}</b> command. Look for the
		<b>a</b> prefix in the status fields. This indicates the port is
		configured for auto-negotiation. Examples are a-full and a-100. If the
		<b>a</b> prefix is not present, the port is manually configured for
		the parameters shown. Examples are full and 100. Issue the <b>show
		configuration [<span style="font-weight:normal; font-style:italic">mod_num</span>]</b> command to view
		the configuration of the switch. On a Catalyst Cisco IOS Software platform,
		issue the command<b> show interfaces
		{<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} status</b>.
		</p>
</li> 
  <li> 
	 <p>How can you tell what your port is capable of? </p> 
	 <p>Issue the <b>show port capabilities
		{<span style="font-weight:normal; font-style:italic">mod_num</span>}|{<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>}</b>
		command. For Catalyst Cisco IOS Software, there is not an equivalent command,
		but you can issue the<b> show interfaces
		{<span style="font-weight:normal; font-style:italic">mod_num/port_num</span>} status</b> command to view
		the speed/duplex settings. </p>
</li> 
  <li> 
	 <p>Why do you receive the error message "Port 1/1 is in auto-sensing
		mode" when you attempt to set the duplex mode?</p> 
	 <p>You receive this error because you need to manually configure the
		speed on a port before you can manually configure the duplex mode.
		</p>
</li> 
  <li> 
	 <p>Why does a port not detect the correct duplex mode when the link
		partner is not configured for auto-negotiation? </p> 
	 <p>The port does not detect it because there is no method available to
		perform this. </p>
</li> 
  <li> 
	 <p>Why is it possible to have link show connected when the two ports
		have different duplex modes configured? </p> 
	 <p>It is possible because the electrical signals the ports use to
		determine if they are connected do not track the status of the duplex modes.
		</p>
</li> 
  <li> 
	 <p>Does the <b>a</b> prefix on the duplex and speed status
		fields always mean the port has auto-negotiated behavior? </p> 
	 <p>No, it means that the port is capable of performing auto-negotiation.
		</p>
</li> 
  <li> 
	 <p>What does the <tt>%CDP-4-DUPLEXMISMATCH:Full/half-duplex
		mismatch detected</tt> message mean? </p> 
	 <p>This means the CDP determines, via a configuration comparison
		dialogue, that a mismatch exists. CDP does not attempt to resolve the mismatch.
		</p>
</li>
</ol>
